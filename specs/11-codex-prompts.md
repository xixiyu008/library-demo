# 11 Codex 常用提示词

## 1. 规格审查提示词

```text
请阅读 specs 目录下所有文档。

暂时不要写代码。

请输出：
1. 系统目标总结
2. 技术栈总结
3. 模块划分总结
4. 数据库表总结
5. API接口总结
6. 可能存在的规格冲突
7. 你建议的开发顺序
```

## 2. 项目骨架生成提示词

```text
请根据 specs 文档生成项目骨架。

要求：
1. 严格遵守 specs/02-architecture-spec.md。
2. 严格遵守 specs/07-coding-rules.md。
3. 只生成目录结构、基础类、配置类。
4. 暂时不要实现复杂业务逻辑。
5. 生成后请说明每个包的职责。
```

## 3. 单模块实现提示词

```text
请实现【模块名称】模块。

要求：
1. 只修改与该模块相关的文件。
2. 不要随意改动已有公共结构。
3. 遵守统一响应格式。
4. 遵守 DTO/VO 分层。
5. 完成后列出新增文件、修改文件、关键业务逻辑。
```

## 4. 代码审查提示词

```text
请对当前代码进行审查。

审查依据：
- specs/01-business-spec.md
- specs/02-architecture-spec.md
- specs/05-api-spec.md
- specs/07-coding-rules.md
- specs/09-review-checklist.md

请输出：
1. 架构违规问题
2. 业务规则遗漏
3. 安全问题
4. API不一致问题
5. 前后端字段不一致问题
6. 建议修复方案
```

## 5. 修复提示词

```text
请根据上一步审查结果修复代码。

要求：
1. 优先修复会导致运行失败的问题。
2. 其次修复业务规则遗漏。
3. 最后修复代码风格问题。
4. 每次修复不要扩大修改范围。
5. 修复完成后重新运行相关测试。
```

## 6. 测试提示词

```text
请根据 specs/10-acceptance-test.md 为当前系统设计测试。

要求：
1. 生成后端接口测试用例。
2. 生成前端手工测试步骤。
3. 检查每个验收测试是否可执行。
4. 如发现系统缺少必要接口，请指出。
```
